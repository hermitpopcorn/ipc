# UDS (Unix Domain Socket) IPC デモ

Unix Domain Socket (UDS) は、同じマシン上のプロセス間通信 (IPC) を行うための仕組み。
ネットワークを通さない TCP/IP ソケットのような感じ。

## 動作原理

1. ハブ側
   - `socket(AF_UNIX, SOCK_STREAM, 0)` でソケットを作成
   - `bind()` でソケットファイルをファイルシステムにバインド
   - `listen()` でクライアント接続を待機

2. クライアント側
   - `socket(AF_UNIX, SOCK_STREAM, 0)` でソケットを作成
   - `connect()` でハブのソケットファイルに接続
   - `send()` / `recv()` でデータ送受信

3. 通信フロー
   - クライアントがハブに接続要求
   - ハブが `accept()` で接続を受け入れ
   - サーバとクライアント間、双方向でデータ送受信

## このデモについて

構成は以下になっている:

- **hub.cpp**: ハブ（メッセージブローカー）
  - クライアントの接続を管理
  - メッセージを宛先のクライアントに転送

- **client.cpp**: クライアント
  - ハブに接続
  - ユーザー入力を受け付け
  - メッセージを送受信
  - 受信メッセージは別スレッドで処理し、表示

### 使用方法

```bash
# ビルド
make

# ハブを起動（ターミナル1）
./hub

# クライアント1を起動（別ターミナルで）
./client c1 c2

# クライアント2を起動（別ターミナルで）
./client c2 c1
```

## 複数クライアント時の設計パターン

UDS は基本的にシンプルだが、複数のクライアント間でのメッセージ通信になると、以下の2つの設計選択肢がある:

1. ハブ方式（このデモで採用）
   - 中央集約型: 1つのハブが全クライアントを管理
   - メリット: スケーラビリティが良い、管理が一元化
   - デメリット: ハブが単一障害点

2. メッシュ方式
   - クライアント間で直接接続（全てのクライアント間でソケットペアを作成）
   - メリット: 単一障害点がない
   - デメリット: クライアント数が増えると管理が複雑、リソース消費が増加（監視すべきソケットが多い）
